## 制御入力の仕組み

Waypointは3つの入力チャンネルでカメラを制御する:

### 1. `mouse` (x, y) — カメラ回転

マウス速度ベクトル。カメラの向きを変える。

- `x > 0`: 右を向く
- `x < 0`: 左を向く
- `y`: 上下の視線変化
- `(0.0, 0.0)`: カメラ方向維持

### 2. `button` (Set[int]) — キーボード入力

256要素のone-hotベクトル。ボタンIDでエンコード。

**重要な制約**: ボタンIDと実際の動作（前進・後退等）のマッピングは公式に文書化されていない。
モデルはゲーム映像で学習しているため、各ゲームのキーバインドに依存する。

HuggingFace公式サンプルで使用されているID:
```python
CtrlInput(button={48, 42}, mouse=[0.4, 0.3])  # 何らかの組み合わせ
CtrlInput(button={95, 32, 105})                # 別の組み合わせ
```

これらはランダムな操作例であり、特定の動きを意図していない。

### 3. `scroll` (int) — スクロールホイール

`-1`, `0`, `1` の3値。sign化してモデルに渡される。

### 制御に関する注意事項

- **「前進」を確実に指示する方法がない**: ボタンIDの意味はモデルの学習元ゲームに依存
- **mouseが最も予測可能**: カメラの回転方向は安定して制御できる
- **テキストプロンプトは補助的**: シーンの雰囲気・スタイルに影響するが、動きの制御力は弱い
- モデルは「テキスト/UIの生成」「人物・動物の顔」が苦手（公式制約）

## デノイズ設定

モデルはDMD (Distribution Matching Distillation) で蒸留されており、少ステップに最適化されている。

- **デフォルト sigmas**: `[1.0, 0.861, 0.729, 0.321, 0.0]` = **4ステップ**
- ステップ数変更は非推奨（CUDAエラー・品質劣化を確認）
- `torch.compile` による高速化が最も効果的

## ボタンIDスキャン結果（2026-03-04）

0-100の全ボタンIDを体系的にテスト（各10フレーム生成、ピクセル変化量で判定）。

### 結果サマリー

| カテゴリ | ボタン数 | 特徴 |
|---|---|---|
| 大きな動き (effect > 10) | 28個 | **全て同方向（R bias）** — 方向の区別なし |
| 中程度 (3-10) | 7個 | 方向ほぼ中立 |
| 微小 (0.5-3) | 62個 | ID 48（Wキー想定）もここ |
| 効果なし (< 0.5) | 4個 | 23, 31, 41, 43 |

### 結論

**ボタンIDによる方向制御（前進・後退・左右）は機能しない。**

- 効果の大きいボタンは全て同じ方向に動く（右バイアス）
- ボタンごとの前進/後退/左右移動の区別は存在しない
- モデルは10,000時間の多種多様なゲーム映像で学習しており、
  特定のキーマッピングは平均化されて失われている

### 方針

- カメラ制御は **mouse velocity のみ** で行う
- `mouse (x, y)` によるカメラ回転は安定して動作する
- 移動（前進/後退）はモデルの自然なドリフトに依存
- UI設計は「自由移動」→「視点回転 + プリセットツアー」に変更
